# 概述

> [!tip] 
>
> 标准文件请参见[Specification of Communication](https://www.autosar.org/fileadmin/standards/R25-11/CP/AUTOSAR_CP_SWS_COM.pdf)

首先得了解一下AUTOSAT面向信号的通信理念，什么个意思呢？这里的信号可以理解成是应用层关心的实际值，比方说电压值，电流值，转速等，这些都是信号，应用层接收和发送信号的时候，它只需要调用相关信号的接口函数触发动作即可，至于这些信号值是通过哪种方式传送和哪种协议解析，应用层根本不关心，这就是AUTOSAR面向信号的通信理念。为了在软件中实现这个理念，Com模块在这其中起着至关重要的作用。

Com 模块位于 RTE 和 PDUR 的中间，其充当 RTE 和 PDUR 通信服务层的接口层，如下图所示：

![Communication stack结构](https://cdn.jsdelivr.net/gh/youShouldTrustMe/MyPictures@main/Images/20260430092221925.png)



其主要功能就是负责为上层提供各种信号的接收和传送，说得简单点一点，就是把很多信号打包成PDU（这里的PDU则是软件协议层面上的一条消息，映射到硬件上，可能是一帧或者几帧，这取决于软件协议组包的方式，比如说TCP\IP协议栈的组包)，然后往各种总线上送，或者从各种总线上接收到PDU后解包成信号提供给上层。 但是众所周知，AUTOSAR就是会把简单的东西搞得及其得复杂，所以Com里面又有一大堆新玩意。

> [!note] 
>
> 首先在介绍Com功能时，就不得不提一下DBC和LDF文件了， 在基于AUTOSAR架构进行通信协议栈的研发时，大家不可避免都会涉及到这两种文件，那么这两种文件到底有什么用呢？在这个世界还没有AUTOSAR之前，其实这两种文件就已经存在且被广泛应用，这两种文件主要定义了CAN报文（DBC文件定义）或者LIN报文(LDF文件定义)的组包方式，除此之外还有不同报文的发送和接收熟悉。
>
> 在AUTOSAR的方法论中，一切文件皆ARXML，所有输入输出文件都是ARXML文件类型，包括与DBC文件和LDF文件起到相同作用的文件，不过当时这两种文件已经特别通用，所以这两种格式的文件也无缝融入了的AUTOSAR的研发过程中，也有很多软件都能实现DBC文件和LDF文件与ARXML文件之间格式的转换。
>
> 关于DBC文件的介绍大家可以参考文章：《[细说DBC(一)——初识](http://www.360doc.com/content/12/0121/07/50927056_1016678962.shtml)》，关于LDF文件的介绍大家可以参考文章[《LIN数据库文件LDF介绍及使用》](https://zhuanlan.zhihu.com/p/541273938)。

 Com 模块主要功能如下：

1. 提供基于信号的发送机制；
   1. 提供不同的发送方式（例如：周期、事件）；
   2. 对报文发送超时进行监测与处理；
   3. 提供发送状态通知机制；
   4. 提供基于信号的发送接口函数。
2. 提供基于信号的接收机制；
   1. 对报文接收超时进行监测与处理；
   2. 提供接收状态通知机制；
   3. 提供基于信号的接收接口函数。
3. 支持信号路由功能；
4. 支持信号滤波功能；
5. 支持长报文（TP）传输；
6. 支持动态信号收发

# 一般功能

## 信号

根据规范，AUTOSAR COM 模块对信号值的处理、数据类型支持以及字节序转换（Endianness Conversion）与符号扩展（Sign Extension）有严格的定义。

1. 信号值的分类

   AUTOSAR COM 模块处理的信号值主要分为以下三类：

   - **初始值 (Init Value)**：用于信号初始化阶段的默认值。
   - **数据无效值 (Data Invalid Value)**：用于标识接收或发送的数据无效。
   - **有效值 (Valid Value)**：这是正常情况下的值，即在初始化阶段后，由模块发送或接收的有效数据。

2. 支持的数据类型

   COM 模块必须支持以下数据类型：

   - **基本整型**：`boolean`, `uint8`, `uint16`, `uint32`, `uint64`, `sint8`, `sint16`, `sint32`, `sint64`。
   - **数组类型**：`uint8[n]`（映射为 `UINT8_N` 或 `UINT8_DYN`）。
   - **浮点型**：`float32`, `float64`。

3. 字节序转换 (Endianness Conversion)

   为了支持不同硬件架构之间的数据交换，COM 模块提供了字节序处理机制：

   - **整型支持**：支持所有有符号和无符号整型数据的字节序转换。
   - **非整型处理**：对于 `uint8[n]` 或 `float32` 等非整型数据，COM 模块会将其视为匹配大小的整型处理；如果配置为 `OPAQUE`（不透明数据），则不对其内容进行解析。
   - **不透明数据 (Opaque Data)**：始终被解释为 `uint8[n]` 并映射到 $n$ 字节大小的信号中，且其字节序配置必须为 `OPAQUE`。
   - **范围扩展**：字节序转换不仅适用于独立信号，也适用于信号组（Signal Groups）中的信号。

4. 符号扩展 (Sign Extension)

   当接收到的有符号信号位宽小于其在软件中定义的变量类型位宽时，COM 模块会执行符号扩展：

   - **处理逻辑**：模块将接收到的数据扩展至 `ComSignalType` 指定的大小，并确保负值能正确映射。
   - **示例**：若接收到一个 10 位的有符号信号并拷贝到 16 位的变量中，如果接收值为 -3（二进制 `11 1111 1101`），扩展后将变为二进制 `1111 1111 1111 1101`。
   - **限制**：COM 模块**不支持**对 `float32` 和 `float64` 类型进行符号扩展。
   - **发送端说明**：发送端不需要执行符号扩展，因为规范保证待发送的值已经是经过符号扩展的。

5. 执行时机与顺序

   转换和扩展的操作在通信流程中有明确的先后顺序：

   - **发送端**：在执行 I-PDU Callout（回调函数）**之前**完成字节序转换。
   - **接收端**：在进行信号过滤（Filtering）和通知检测（Notification detection）**之前**，必须先完成符号扩展和字节序转换。
   - **浮点数限制**：对于 `FLOAT32` 或 `FLOAT64` 类型，除了字节序转换外，COM 模块不支持任何复杂的格式转换、规范化（归一化）或偏置值（Bias）调整。

## 过滤

**信号过滤（Signal Filtering）**主要目的是决定接收到的数据是否应当被进一步处理，或者用于评估发送端的传输模式。

1. 过滤的基本原则

   - **计算结果**：COM 模块必须将每个过滤条件计算为 `true`（通过）或 `false`（被过滤）。
   - **接收端过滤**：COM 模块仅在接收端过滤掉信号。如果过滤条件评估为 `false`，该信号将被丢弃且不再处理。
   - **信号组处理**：如果一个“组信号（Group Signal）”被过滤掉，COM 模块将丢弃整个信号组。
   - **发送端行为**：在发送端，过滤机制仅用于评估**传输模式条件（Transmission Mode Conditions, TMC）**，而不会真正过滤掉信号（即信号始终会发送，只是可能改变发送频率或模式）。

2. 支持的过滤算法 (`ComFilterAlgorithm`)

   为了降低复杂度，AUTOSAR 仅支持 OSEK COM中的一部分算法：

   - **ALWAYS / NEVER**：始终通过或始终过滤。
   - **MASKED_NEW_EQUALS_X / MASKED_NEW_DIFFERS_X**：新值经过掩码处理后，是否等于/不等于预设值 X。
   - **MASKED_NEW_DIFFERS_MASKED_OLD**：新值与旧值分别经过掩码处理后，两者是否不同。
   - **NEW_IS_WITHIN / NEW_IS_OUTSIDE**：新值是否在设定的范围内/范围外。
   - **ONE_EVERY_N**：每 $N$ 个信号中通过一个。

3. 关键变量管理：新值与旧值

   过滤通常涉及 `new_value`（当前接收/发送的值）和 `old_value`（上一个通过过滤的值）的对比。

   - **启动初始化**：在启动时，`old_value` 被设置为信号的初始值（`ComSignalInitValue`）。
   - **首次更新前**：在应用层通过 API 更新信号值之前，`new_value` 也使用初始值进行过滤计算。
   - **旧值更新逻辑**：
     - 只有当过滤结果为 `true` 时，当前信号值才会存入 `old_value`。
     - 如果信号被过滤（结果为 `false`），`old_value` 保持不变，不会被当前值覆盖。

4. 特殊算法逻辑

   - **位宽限制**：对于 `MASKED` 或范围类过滤，计算时仅使用信号最低有效位（Least Significant Bits）对应长度的过滤参数。
   - **ONE_EVERY_N (周期性过滤)**：
     - **OFFSET**：当计数器（OCCURRENCE）等于偏移量时，返回 `true`。
     - **PERIOD**：当计数器达到周期值时，计数器归零。
     - **处理流**：每次过滤处理后，计数器自增。
   - **超时后的特殊处理**：对于 `MASKED_NEW_DIFFERS_MASKED_OLD` 算法，如果发生了接收超时监测（Deadline Monitoring）超时，超时后收到的**第一帧数据**必须强制视为“通过过滤”，无论其值如何。

5. 数据类型支持

   COM 模块必须确保上述所有的过滤机制同样支持**有符号数据类型（Signed Data Types）**。

## 信号路由

AUTOSAR COM 模块集成的 **信号网关 (Signal Gateway)** 是实现跨通信链路信号分发的关键组件。它允许信号或信号组以 $1:n$ 的方式进行高效转发。

1. 静态配置与路由关系

   - **配置方式**：信号网关的路由关系不是动态生成的，而是通过 `ComGwMapping` 配置容器进行**静态配置**。
   - **角色切换**：对于被配置为网关源的信号或信号组，集成网关首先充当**接收者**；一旦接收完成，它立即转变为**发送者**执行转发。
   - **处理一致性**：无论是网关转发的信号还是由软件组件（SW-C）发送的信号，其信号处理流程（如过滤、转换等）在逻辑上是完全一致的。

2. 性能优化原则

   由于路由关系是静态确定的，规范允许在不改变逻辑行为的前提下进行性能优化：

   - **阶段跳过**：在特定配置下，网关可以跳过某些处理阶段（例如，当源总线和目标总线大小端一致时，跳过**字节序转换**）以提高执行效率。
   - **约束**：任何优化都不得导致目标总线上的数据表示或逻辑行为与标准流程产生偏差。

3. 信号组一致性与阴影缓存 (Shadow Buffer)

   - **并发处理**：一个信号可以同时被本地 ECU 接收供应用层使用，并作为网关源触发转发动作。
   - **一致性保障**：虽然规范中提到的“阴影缓存”通常展示在由 RTE 接收的信号组路径上，但集成网关必须确保在处理转发的信号组时维持数据的一致性。规范并未强制规定实现一致性的具体技术手段，给予了开发者实现上的灵活性。

4. 可伸缩性 (Scalability)

   **零成本裁剪**：如果 ECU 项目中不需要信号路由功能，AUTOSAR COM 模块的信号网关部分应当能够完全剥离，不占用任何系统资源或代码空间。

# 常规操作

## 启动行为



1. I-PDU 初始化序列

   在执行 `Com_Init` 时，每个 I-PDU 的初始化遵循严谨的“先整体后局部”的顺序：

   1. **逐字节默认填充 (Byte-wise)**：

      首先使用配置的 `ComTxIPduUnusedAreasDefault` 值对整个 I-PDU 内存进行逐字节填充。这确保了 PDU 中未定义的区域或填充位（Padding bits）具有确定的初始状态。

   2. **逐位信号初始化 (Bit-wise)**：

      随后，根据每个信号定义的初始值（`ComSignalInitValue`）对 I-PDU 进行位级覆盖。

   3. **更新位处理 (Update-bits)**：

      同时根据信号的配置初始化更新位（Update-bits）的状态。

2. 补充初始化逻辑

   结合前后的规范要求，启动阶段还涉及以下关键动作：

   - **过滤器状态初始化**：

     在启动期间，COM 模块会将每个信号的过滤机制中的 `old_value` 设置为该信号的 `ComSignalInitValue`。

   - **新值预设**：

     在应用层通过发送 API 更新信号之前，过滤机制将始终使用 `ComSignalInitValue` 作为 `new_value` 进行计算。

   - **符号扩展**：

     COM 模块会自动将配置的初始值（`ComSignalInitValue`）扩展至该信号类型（`ComSignalType`）所占用的完整位宽。

   - **ONE_EVERY_N 计数器**：

     如果 I-PDU 组启动时设置了初始化参数，`ONE_EVERY_N` 过滤算法中的 `OCCURRENCE` 计数器将被清零。

这种“先填充默认值再覆盖信号值”的设计，保证了 I-PDU 在总线上首次发出时，即使包含复杂的对齐或保留位，其数据布局也是完全可控且符合预期的。

## 前提条件

1. C 启动代码的职责 (Start-up Code)

   在调用任何 AUTOSAR COM 服务之前，C 启动代码（通常在 IAR 等编译器中被称为 `__iar_program_start` 或类似的 `reset_handler`）必须完成以下任务：

   - **变量初始化**：将具有显式初始值的全局变量和静态变量从 Flash 拷贝到 RAM。
   - **零初始化 (BSS)**：将未显式初始化的全局/静态变量清零。
   - **栈/堆设置**：配置 CPU 寄存器（如 SP），为 C 函数的调用准备好内存环境。

2. 对 COM 模块的影响

   COM 模块的行为高度依赖于这些内存的初始状态：

   - **信号初始值引用**：COM 模块在执行 `Com_Init` 时，需要读取配置在 Flash 中的信号初始值（`ComSignalInitValue`）并覆盖到 I-PDU 的缓存区中。
   - **过滤器状态维护**：过滤机制依赖于 RAM 中的 `old_value` 变量，该变量在启动时必须被正确初始化为 `ComSignalInitValue`。
   - **数据一致性**：如果 COM 服务的调用（如 `Com_ReceiveSignal`）发生在 C 初始化代码执行之前，可能会读取到随机的内存垃圾数据，导致系统行为异常。

3. I-PDU 初始化流程回顾

   一旦 C 运行环境就绪，调用 `Com_Init` 将会按照以下物理顺序初始化通信缓冲区：

   - **全字节填充**：首先使用 `ComTxIPduUnusedAreasDefault` 逐字节填充整个 I-PDU。
   - **位级覆盖**：根据信号定义，将 `ComSignalInitValue` 以位偏移的方式写入 PDU。
   - **符号扩展处理**：在写入过程中，初始值会被自动扩展至信号定义的类型位宽（`ComSignalType`）。

💡 实战建议 

在您的开发环境中，这种“前提条件”通常体现在以下工程实践中：

- **执行顺序**：确保 `Com_Init` 的调用位于硬件底层初始化（如时钟、GPIO）之后，且在所有周期性任务（如 OS 调度或定时器中断）启动**之前**。
- **内存分配**：在 IAR 的 `.icf` 链接文件中，确保 COM 模块的配置常量（`.rodata`）和运行缓冲区（`.bss` 或 `.data`）被分配在正确的地址区间。
- **调试检查**：如果您发现 `Com_Init` 之后信号值不正确，请检查是否在 C 启动代码执行过程中由于 Watchdog 复位导致内存拷贝被中断。

## 初始化

在 AUTOSAR 通信栈中，`Com_Init` 的初始化逻辑不仅是对内存的简单擦除，而是一套确保通信确定性的多层填充机制。

1. 结构体与内部数据初始化

   **补充初始化**：`Com_Init` 必须初始化所有未被 C 启动代码（Start-up code）处理的内部数据（例如内部维护的状态结构体 C-structs）。这确保了 COM 模块的运行环境在逻辑上是完整的。

2. I-PDU 未使用区域的填充

   - **确定性填充**：对于 I-PDU 中未被任何信号定义的“空白”区域，COM 模块必须使用配置参数 `ComTxIPduUnusedAreasDefault`（例如 `0xFF`）进行填充。
   - **物理意义**：这能防止由于内存随机值导致的总线报文在未使用位上出现跳变，有利于总线负载监测和数据对齐。

3. 信号与更新位（Update-Bits）的初始化

   - **信号位宽匹配**：对于 $n$ 位大小的信号，COM 模块在发送端和接收端初始化时，必须使用 `ComSignalInitValue` 的**低 $n$ 位**进行填充。
   - **无效值关联**：初始值 `ComSignalInitValue` 可以与 `ComSignalDataInvalidValue` 设置为相同的值，但每个信号可以独立配置。
   - **更新位清零**：在初始化期间，所有的更新位（Update-bits）必须被**强制清零**。这保证了在应用层显式更新信号之前，接收端不会误认为收到了新数据。

4. I-PDU 组的默认状态

   - **默认停止**：默认情况下，所有的 I-PDU 组在初始化后都处于 **Stopped（停止）** 状态。
   - **手动启动**：调用 `Com_Init` **不会自动启动**任何 I-PDU 组。这意味着在 `Com_Init` 执行完后，系统仍不会在总线上发送或接收数据，直到应用层通过 `Com_IpduGroupStart` 显式开启特定的组。

💡 嵌入式实战提示 

在 IAR 环境下调试时，您可以观察 I-PDU 缓存区的内存变化：

1. **Step 1**: `Com_Init` 开始，缓冲区被全字节填充为 `ComTxIPduUnusedAreasDefault`。
2. **Step 2**: 对应的信号位（Signals）被改写为 `ComSignalInitValue`。
3. **Step 3**: 所有的更新位所在位置被置 0。

## 去初始化

`Com_DeInit` 是 AUTOSAR COM 模块用于**去初始化**的核心 API。它的调用标志着通信层进入非活动状态。

1. 通信能力的终止

   - **停止所有通信**：执行 `Com_DeInit` 后，通过 AUTOSAR COM 模块进行的所有通信都将被终止。
   - **状态强制转换**：所有原本处于启动（Started）状态的 I-PDU 组都会被强制停止（Stopped）。

2. 内部逻辑与

   该要求通常涵盖：

   - **资源释放**：模块应将内部状态变量、定时器和缓冲区恢复到未定义或默认的去初始化状态。
   - **API 保护**：在 `Com_DeInit` 执行之后、下一次 `Com_Init` 执行之前，除了 `Com_GetStatus` 等少数管理类 API 外，调用其他通信类 API（如 `Com_SendSignal`）通常会返回错误（如 `COM_E_UNINIT`）或被 DET 模块捕获。

3. 与启动行为的对比

   正如 `Com_Init` 确保了系统从确定的初始值开始运行，`Com_DeInit` 则确保了系统在关闭或进入低功耗模式前能优雅地停止所有挂起的传输任务。

💡 实战建议

在您使用 IAR 开发嵌入式系统时，处理 `Com_DeInit` 需要注意以下几点：

- **硬件协同**：在调用 `Com_DeInit` 前，通常建议先关闭底层的 CAN 控制器（通过 `CanIf` 或 `Can` 模块），以防止 COM 层停止后底层驱动仍尝试访问已释放或已禁用的内存缓冲区。
- **低功耗设计**：如果您的项目需要进入深睡眠模式（Deep Sleep），`Com_DeInit` 是清理内存状态、防止静态电流异常的关键步骤。
- **重新初始化**：如果您在去初始化后需要重新启动通信，必须重新调用 `Com_Init`。此时，所有的信号都会根据 `ComSignalInitValue` 重新加载。

## 通信模式

1. 传输模式的切换机制

   AUTOSAR COM 允许为一个 I-PDU 配置两种不同的传输模式。 这种机制称为**传输模式选择 (Transmission Mode Selection, TMS)**：

   - **切换触发**：模块会根据 I-PDU 中包含的信号是否满足特定的过滤条件（即 **TMC - Transmission Mode Condition**）来评估当前应使用哪种模式。
   - **模式定义**：通常分为 `ComTxModeTrue` 和 `ComTxModeFalse`。

2. 核心传输模式分类

   根据信号变化的触发方式，主要分为以下几种基本模式：

   | **模式**            | **描述**                                                     |
   | ------------------- | ------------------------------------------------------------ |
   | **DIRECT (直接)**   | 当信号通过 API（如 `Com_SendSignal`）更新时，立即触发 I-PDU 发送。 |
   | **PERIODIC (周期)** | 无论信号值是否变化，I-PDU 都会按照配置的固定周期（`ComTxModeTimePeriod`）循环发送。 |
   | **MIXED (混合)**    | 结合了周期和直接触发。信号更新时立即发送，且在无更新时保持周期性发送。 |
   | **NONE (无)**       | COM 模块不主动触发发送，通常用于仅由 `Com_TriggerTransmit` 请求的底层驱动。 |

3. 信号复制 (Replication)

   通常用于提高关键数据的容错性或满足特定的功能安全需求，允许在同一个 I-PDU 中多次包含相同或相关的信号数据。

4. 数据一致性保障

   在处理这些传输模式时，COM 模块会通过以下流程确保数据正确性：

   - **字节序转换**：在发送端执行 I-PDU Callout 之前完成。
   - **符号扩展**：在接收端处理过滤和通知之前，先完成字节序转换和符号扩展。
   - **过滤评估**：过滤结果决定了 `old_value` 的更新以及是否触发传输模式的切换。

### I-PDU传输

###### 信号传输

AUTOSAR COM 模块中**信号传输属性 (Transfer Property)** 与 **I-PDU 传输模式 (Transmission Mode)** 之间的协作。它们共同决定了当你调用 `Com_SendSignal` 时，总线上是否会立即出现报文。

1. 信号传输属性 (Transfer Properties)

   传输属性决定了信号的“更新”行为是否具有触发 I-PDU 发送的效力。

   | **传输属性**                               | **触发条件**                         | **发送次数**                   |
   | ------------------------------------------ | ------------------------------------ | ------------------------------ |
   | **TRIGGERED**                              | 只要调用 `Com_SendSignal`。          | $NumberOfRepetitions + 1$ 次。 |
   | **TRIGGERED_WITHOUT_REPETITION**           | 只要调用 `Com_SendSignal`。          | 仅 1 次。                      |
   | **TRIGGERED_ON_CHANGE**                    | 调用 API 且信号值/长度发生**变化**。 | $NumberOfRepetitions + 1$ 次。 |
   | **TRIGGERED_ON_CHANGE_WITHOUT_REPETITION** | 调用 API 且信号值/长度发生**变化**。 | 仅 1 次。                      |

   > **注意**：`ON_CHANGE` 类型的属性不仅适用于数值（如 `uint32`），也适用于**字节数组**（如 `UINT8_N`）。COM 模块会将当前值与本地存储的“旧值”或“初始值”进行对比。

2. 与传输模式的配合逻辑

   这些触发行为主要作用于配置为 **DIRECT** 或 **MIXED** 模式的 I-PDU。

   - **立即性**：一旦满足触发条件，COM 模块必须立即启动发送流程，最迟不得晚于下一个 `Com_MainFunction` 周期。
   - **重复发送机制**：配置了 `NumberOfRepetitions` 的信号在触发时会启动多次发送（通常用于增加关键信号的抗干扰性），这也是为什么规范里写的是 $N+1$。
   - **NONE 模式**：如果 I-PDU 的模式设为 `NONE`，COM 模块**绝不会**主动发起发送请求。它只能由底层驱动（如 LIN/FlexRay 调度表）通过 `Com_TriggerTransmit` 动。

3. PDU 级触发：Com_TriggerTransmit

   这是一种“被动”发送机制，常用于 LIN 或 FlexRay 总线：

   - **跨总线兼容**：无论 I-PDU 配置为何种模式（DIRECT, PERIODIC 等），底层模块（如 PduR）都可以调用 `Com_TriggerTransmit` 来拉取当前 PDU 的最新数据。
   - **信号“搭便车”**：如果一个 I-PDU 中包含多个信号，只要其中一个信号（带有 `TRIGGERED` 属性）触发了发送，或者周期到了，PDU 中的**所有**挂起（Pending）信号都会被一同发出。





































# 报文的发送

首先得强调一点就是信号和报文（PDU）的关系是一条报文会包含一个或者多个消息。 报文的发送则是由报文传输特性及信号传输特性共同决定的。首先得了解一下报文和信号各有哪些传输特性。

报文的传输特性包括：

1. **NONE**：该类型的报文不会被发送；
2. **PERIODIC**：该类型报文的发送方式为周期发送，不会随发送信号的传输特性改变而改变；
3. **DIRECT**：该类型报文的发送方式完全由发送信号的传输特性决定；
4. **MIXED**：该类型报文的发送方式由报文的传输特性及发送信号的传输特性综合决定。

信号的传输特性包括：

1. **PENDING**：该类型信号的发送，是依据其所在报文的发送类型；
2. **TRIGGERED_WITHOUT_REPETITION**：每当写该信号时，触发一次发送；
3. **TRIGGERED**：每当写该信号时，触发多次发送；
4. **TRIGGERED_ON_CHANGE_WITHOUT_REPETITION**：每当该信号的值发送变化时，触发一次发送；
5. **TRIGGERED_ON_CHANGE**：每当该信号的值发送变化时，触发多次发送。

> [!tip]  
>
> **注1**：触发多次发送时，发送次数由该信号所在报文的 ComTxModeNumberOfRepetitions属性决定。多次发送的周期，由该信号所在报文的 ComTxModeRepetitionPeriod 属性决定。
>
> **注 2**：触发发送时，并非立即发送，而是需要等到下一个 Com_MainFunctionTx 的调度周期，才开始进行发送。

# TP报文

需要使用传输协议进行传输的报文（通常是因为报文长度较大，单帧无法进行传输），需要配置为 TP 报文。对 COM 模块而言，TP 报文与非 TP 报文的区别主要包括：

1. 与 PDUR 交互的接口不同；
2. 由于 TP 报文需要多个周期才能发送成功，为保证数据一致性，TP 报文发送过程中，禁止更新发送信号，直至报文发送成功或超时；
3. 多帧报文在接收过程中，可能出现丢帧、超时或其他错误情况，在接收错误的情况下，COM 将丢弃未成功接收的报文，并且调用 Com_CbkInv 通知用户。

# 功能组

用户可以对不同的 PDU 进行分组，位于同一组内的 PDU，可以同步的进行启动或停止的操作。PDU 组的功能默认是开启的，默认的分组规则是，每个通道单独分组，每个通道分为发送和接收两个功能组。

位于功能组内的报文，在初始化后，处于关闭的状态。用户需要调用相关接口，启动该组后，组内的报文才可以正常的收发。不属于任何功能组的报文，在初始化后，处于开启的状态，可以正常进行收发。

同一报文可以属于多个功能组，称为关联组。开启一个功能组时，只开启属于本功能组的所有报文。关闭一个组时，如果该功能组有关联组，则会关闭本组及所有关联组内的所有报文。不同的功能组也可以组成一个新的功能组，这称为多级的功能组。

# 超时监测

超时属性是属于信号属性，需要通过配置工具进行配置。但实际上，信号是不能单独进行传输的，必须以报文为单位进行传输，因此，COM 的超时监测实际是以报文为单位进行的。同一条报文中，不同信号可以配置不同超时值。报文的超时值等于信号中最小的超时值，报文中各信号的实际超时值等于报文的超时值。

发送超时是指从启动报文发送到发送确认之间的时间。如果信号的传输特性中配置了重发机制（Repetition），那么在发送超时的时间内，必须将全部重发次数发完，否则 COM 会上报发送超时（在这里简单得解释一下Repetition机制，就是有的事件型报文触发之后可以配置它在短时间内发送多次，然后这里得超时时间则指的是在超时时间内必须将多次全部发送完成）。超时后，未完成的发送将被丢弃。COM 监测到发送超时事件后，会调用 Com_CbkTxTOut 通知用户。

接收超时是指在规定时间内没有接收到报文。接收超时有两种超时：1. 首次接收超时（ComFirstTimeout），仅在下列场景使用： 初始化后；报文所在的功能组重新启动； 报文所在的功能组的接收超时重新启动。2. 正常接收超时（ComTimeout），正常情况下的超时监测。首次超时时间一般设置的比较长，因为初始化时，总线上各控制器的初始化时间可能比较长。COM 监测到接收超时事件后，会调用 Com_CbkRxTOut 通知用户，并会根据配置，执行不同的动作：1. REPLACE：使用初始值替换当前的信号值；2. NONE：无特殊动作。

# 最小延迟时间

为防止总线的负载率过高，用户可以为发送的 Pdu 配置最小延迟时间。配置了最小延迟时间后，在该时间内，最多只能有 1 帧报文发送到总线上。如果在该时间内有多于 1 次发送请求，则后续的请求会被退后到最小延迟时间之后发送。该时间仅对相同 PduId 的发送报文起作用，不同 PduId 的报文之间无最小延迟时间的限制。

1.
